1. Introduction (Pages 1-3)
---------------------------

1.1 Proposed System and Objectives
  • Deliver a unified credential/document vault with AI assistive tooling.
  • Provide admins with automated sentiment dashboards and AI summaries.
  • Maintain on-prem hosting requirements (XAMPP) to avoid cloud lock-in.
  • Objectives:
      - Encrypt and store passwords, cards, and documents safely.
      - Auto-classify user feedback (positive/negative/needs review/suggestion).
      - Extract document semantics (type, expiry, summary) using OCR + LLMs.
      - Support dark/light themes, screenshot permissions, privacy compliance.

1.2 Core Components
  • Android Client (Kotlin, MVVM, Room, Coroutines, ML Kit, PDFBox).
  • PHP/Apache backend with REST-style endpoints in `Admin/dashboard/api`.
  • MySQL schema (`safepassage.sql`) with tables: users, passwords, cards, documents, feedbacks, pins, utilities, categories, admins.
  • AI connectors: `Admin/dashboard/includes/openai_client.php`, Android `DocumentAIAnalyzer/Orchestrator`.
  • Security utilities: `CryptographyManager`, `PinViewModel`, `SetPinActivity`.

1.3 Minimum / Maximum Requirements
  • Server: Windows 10/11, XAMPP (Apache+MySQL), PHP >=7.4, 4 GB RAM minimum.
  • Client: Android 8+, 2 GB RAM minimum (recommended 4 GB), active internet/WiFi or USB debugging.
  • Development: Android Studio, Java JDK 11+, ADB, Python optional for scripts.

1.4 Advantages and Limitations
  Advantages:
    - AI-driven insights reduce manual analysis.
    - End-to-end encryption using Android Keystore.
    - Modular MVVM architecture simplifies maintenance.
    - Works entirely on LAN (no public hosting required).
  Limitations:
    - Requires XAMPP instance running continuously.
    - AI features need OpenRouter/OpenAI key and internet connectivity.
    - Currently Android-only client; no iOS/web app.
    - Item-level locks pending remote sync support.

SafePassage / WallWonders – Complete Technical Reference
=========================================================

1. Project Overview
-------------------
SafePassage is a hybrid Android + PHP/MySQL platform that combines a secure password/document vault with AI-powered feedback analysis and document intelligence. The “WallWonders” feature set layers AI models (via OpenRouter/OpenAI) on top of traditional credential management to provide auto-summaries, expiry detection, and health insights. The stack is designed for on-premise hosting via XAMPP, making it deployable without public cloud dependencies.

2. Technology Stack
-------------------
Frontend (Mobile):
  • Android app (Kotlin, AndroidX, Material Components)
  • Room persistence (Password, Card, Document, Utilities, Pin entities)
  • Coroutines + LiveData + ViewModel architecture
  • ML Kit Text Recognition, PDFBox-Android for OCR/PDF parsing

Backend (Server):
  • PHP 8 / Apache (XAMPP)
  • MySQL 5.7+ database (schema in `safepassage.sql`)
  • REST-like APIs under `Admin/dashboard/api`
  • OpenRouter/OpenAI integration via `OpenAIClient` wrapper (Sentiment, document AI, debug tooling)

Shared / Utilities:
  • CryptographyManager (AES/GCM keys per user via Android Keystore)
  • SessionManager (SharedPreferences-based user session caching)
  • IpConfig + ApiUrlBuilder for dynamic endpoint resolution

4. Backend Schemas & Tables
---------------------------
Primary database: `safepassage`

Key tables (from `safepassage.sql` / `Admin/dashboard/sql/complete_schema.sql`):
  • `users` – core user profile (user_id, user_name, email, password hash, timestamps)
  • `passwords` – saved login entries (pass_id, user_id, title, username/email/phone, encrypted password/note, cat_id, timestamps)
  • `cards` – stored card details (car_id, user_id, title, card_holder_name, card_number, expiration_date, cvv, pin, note)
  • `documents` – uploaded files (document_id, user_id, title, file metadata, AI columns: ai_status, ai_summary, ai_expiry_date, etc.)
  • `feedbacks` – user feedback records enhanced with sentiment fields (label, confidence, reason, keyword counts)
  • `pins` – application-level PIN hashes (user_id, pin_hash/encrypted_pin + created_at)
  • `utilities` – per-user feature toggles (lock app, screenshot permission, DB sync placeholder)
  • `categories`, `admins`, `installations_state`, auxiliary indexes

Every table is keyed by SAFE UUID or auto-increment ID, and foreign keys enforce cascade deletes for user-owned objects.

5. Backend APIs (Selected)
--------------------------
Location: `Admin/dashboard/api`

  • `register_pin.php` / `verify_pin.php` / `check_pin_status.php`
        - Handle storing/verifying application PIN hash per user.
        - Expect JSON body { userId, pin }.
  • `get_feedbacks_page_data.php`
        - Aggregates feedback entries + sentiment counts for dashboard filters.
  • `generate_ai_summary.php` (removed per user request) / `openai_client` debug endpoints.
  • `privacy_policy.html` (served statically for in-app browser access).

All endpoints rely on `config/db.php` for PDO connection with environment overrides (DB_HOST, DB_NAME, etc.).

6. OpenAI / OpenRouter Integration
-----------------------------------
Files:
  • `Admin/dashboard/includes/openai_client.php`
  • `app/src/main/java/.../DocumentAIAnalyzer.kt`
  • `app/src/main/java/.../DocumentAIOrchestrator.kt`

Key behaviors:
  • Dashboard uses `OpenAIClient::classifySentiment` to process pending feedback, store label + category.
  • Analyzer builds a structured prompt describing expected JSON (document_type, expiry, summary, confidence).
  • Orchestrator pipeline:
        1. Extract text (PDFBox or ML Kit).
        2. Call LLM via OpenRouter with BuildConfig API key.
        3. Persist AI metadata and update Room entity state (PENDING → SUCCESS/FAILED).

7. Android Architecture Details
-------------------------------
Entities & DAOs:
  • `Password`, `Card`, `Document`, `Utilities`, `Pin`.
  • Each DAO exposes LiveData queries + update operations (search, sort, etc.).

Repositories:
  • `PasswordRepository`, `CardRepository`, `DocumentRepository` wrap DAO calls for ViewModels.

ViewModels:
  • `PasswordViewModel`, `CardViewModel`, `DocumentViewModel`, `UtilitiesViewModel`, `PinViewModel`.
  • Manage coroutine scopes for insert/update/delete, provide LiveData to UI.

UI Layer:
  • Activities: `MainActivity`, `Login_Screen`, `StartActivity`, `ViewDocumentActivity`, `ViewLoginActivity`, `SetPinActivity`, etc.
  • Fragments: `HomeFragment`, `SettingFragment`, `AddFragment` bottom sheet.
  • Adapters: `PasswordAdapter`, `CardAdapter`, `DocumentAdapter`, `UnifiedAdapter`.
        - DocumentAdapter shows AI status badge and opens dialog with type/expiry/summary.
        - UnifiedAdapter merges passwords + documents (cards accessible via filter).

Utilities:
  • `CryptographyManager` – AES/GCM encrypt/decrypt using per-user keystore alias.
  • `SessionManager` – persists last logged-in user ID for cross-activity retrieval.
  • `ThemeManager` – handles dark/light mode toggle persisted in SharedPreferences.
  • `Utils.pickDocument` – content picker for AddDocument workflow.

8. Security Considerations
--------------------------
  • All sensitive fields (passwords, card numbers, document AI summary?) are encrypted before Room persistence.
  • PINs stored locally are encrypted; remote PINs hashed via bcrypt.
  • Documents saved to internal storage path returned by `DocumentManager.copyFileToInternalStorage`, accessible via FileProvider only.
  • Settings toggle for allowing screenshots (default off).
  • Planned “item lock” functionality extends Pin architecture to per-document/password locks.

9. AI Feedback & Dashboard Features
-----------------------------------
  • `Admin/dashboard/pages/feedbacks.php` renders:
        - Sentiment filter chips (Positive, Negative, Neutral, Needs Review, Suggestions).
        - Table displays label, confidence, reason, category.
        - Auto-refresh JS polls API to keep counts consistent.
  • `feedback_sentiment_service.php`:
        - Ensures DB schema contains `sentiment_category`.
        - Processes pending records by calling OpenAI client.
  • Debug page `Admin/dashboard/debug/test_openai_client.php` allows manual text classification.

10. Document AI UX
------------------
Android UI components:
  • `document_item_layout.xml` – card with file icon, metadata, AI badges, “View AI details” button, delete icon.
  • `dialog_ai_details.xml` – gradient header, status chip, rows for Type/Expiry/Summary, responsive width calculation.
  • `DocumentAdapter.showAiDetailsDialog` – inflates dialog, populates fields, handles fallback statuses.

AI pipeline modules:
  • `DocumentTextExtractor` – handles pdf/docx plain text extraction, image OCR, asynchronous operations.
  • `DocumentAIResult` – data class { documentType, expiryDate, summary, confidence }.
  • `DocumentAIAnalyzer` – builds prompt, calls OpenRouter, parses JSON.
  • `DocumentAIOrchestrator` – orchestrates extraction + analysis, updates Room entity fields, handles retries/failure reasons.

11. Settings & Utilities
------------------------
`SettingFragment`:
  • Displays user info (from `UserViewModel`), feedback submission entry point.
  • Theme toggle (replaces old DB sync switch) uses `ThemeManager`.
  • Screenshot toggle persists to Utilities table.
  • Privacy policy button fetches hosted URL from Firebase Realtime Database.

`ThemeManager`:
  • SharedPreferences store `theme_mode`.
  • Applies AppCompatDelegate mode in `MainActivity`, `StartActivity`, `Login_Screen`.

12. Deployment & Setup Notes
----------------------------
Refer to `README.md` for full instructions. Highlights:
  • Bring up XAMPP (Apache + MySQL), import `safepassage.sql`.
  • Update `IpConfig.CURRENT_IP` when network changes.
  • For USB debugging, run `adb reverse tcp:80 tcp:80` and set IP to `127.0.0.1`.
  • Ensure firewall allows port 80; test `http://<IP>/SafePassage/Admin/...`.
  • Default admin credentials: admin/admin.

13. Troubleshooting Checklist
-----------------------------
  • Connection timeout → confirm IP, firewall, XAMPP, uninstall/reinstall app.
  • DB errors → verify MySQL running, tables exist, re-import schema.
  • Admin dashboard blank → check Apache logs, correct project path (`D:\xampp\htdocs\SafePassage\`).
  • AI errors → test via debug page, verify OpenRouter key in `openai_credentials.php` or BuildConfig.

  • MySQL Workbench or phpMyAdmin (for database management)
  • ADB (Android Debug Bridge) for USB debugging

16.5 Network Requirements
--------------------------
For Local Development:
  • Same WiFi network for server and Android device, OR
  • USB debugging with port forwarding: `adb reverse tcp:80 tcp:80`
  • Firewall configured to allow port 80 (HTTP) and 3306 (MySQL, if remote access needed)

For Production/Deployment:
  • Static IP address or domain name for server
  • Port forwarding on router (if behind NAT)
  • SSL/TLS certificate (recommended for HTTPS)
  • Firewall rules for HTTP (80) and HTTPS (443)

16.6 Performance Benchmarks (Reference)

  • API response time: < 500ms for simple queries

  • API response time: < 200ms for simple queries
  • Document upload: < 1s for files up to 5 MB

Android App Performance (Minimum):

  • Document list load: < 2s

Android App Performance (Recommended):
  • App launch: < 1.5s
